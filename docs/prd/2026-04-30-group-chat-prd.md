# OpenTeam 群聊功能 PRD

## 0. 设计概念图

![OpenTeam 群聊功能设计概念图](../assets/group-chat-ui-concept.png)

这张设计图展示了 OpenTeam 群聊功能第一版的整体产品形态：左侧是类似微信的群聊列表，用户可以在多个 AI 群聊之间切换；中间是当前群聊的任务和多角色回复，支持引用某个角色观点，也支持在输入时通过 `@` 选择指定角色；右侧是当前群聊成员状态和角色配置，角色配置第一版开放角色模板选择、角色名称/描述/提示词编辑和初始化/重新初始化入口，不提供模型、temperature 等参数设置。

左下角的“设置”入口通过弹窗承载通用配置、存储空间和关于信息，避免把全局设置放在右侧角色区。右侧区域只服务当前群聊和当前角色，重点呈现角色状态、恢复会话、提示词编辑等与讨论直接相关的能力。整体 UI 使用 Tailwind CSS 实现，目标是做成专业、高质感、适合长期使用的 AI 协作工作台，而不是普通插件弹窗。

## 1. 背景

OpenTeam 当前已经验证了“一个插件团队页中承载多个 Gemini iframe”的技术路线。每个角色可以绑定一个独立 Gemini 会话，用户可以在统一页面中向角色发送消息，并把回复汇总到群聊面板。

下一阶段要把这个能力从“多角色并行问答”升级为更完整的“AI 群聊工作台”。用户可以创建群聊、选择或创建角色、初始化所有角色，然后围绕一个任务进行多轮讨论。系统需要支持两种讨论模式：

- 独立专家模式：角色彼此隔离，默认独立回答。
- 协作群聊模式：角色知道群聊成员，并在后续轮次中参考其他角色发言。

## 2. 产品目标

### 2.1 核心目标

- 让用户可以创建一个 AI 群聊，并在群聊内组织多个角色参与讨论。
- 让用户可以创建多个群聊，并像微信会话列表一样在左侧切换。
- 让群聊、角色、消息、引用关系和上下文同步状态全部持久化到本地，刷新页面或重新打开插件后仍然保留。
- 让用户可以快速创建角色模板，并在创建群聊或添加角色时从模板库选择角色。
- 让所有角色在群聊初始化时打开并准备好对应 Gemini iframe。
- 让用户无 `@` 发言时，默认发给群聊中的全部角色。
- 让用户通过 `@角色` 定向提问一个或多个角色。
- 让用户可以引用群聊中的某条消息，再发给指定角色进行回应。
- 支持独立专家模式和协作群聊模式，满足“独立对比”和“团队讨论”两类场景。

### 2.2 非目标

第一版不做：

- 多用户实时协作。
- 云端同步。
- 角色市场。
- 自动无限轮讨论。
- AI 自动决定下一个发言者。
- 复杂权限系统。
- 跨浏览器支持。

## 3. 用户画像与使用场景

### 3.1 目标用户

- 产品经理：希望让多个角色从产品、技术、市场、风险角度共同分析问题。
- 创业者：希望模拟小团队讨论一个项目想法。
- 内容创作者：希望让不同视角的 AI 共同产出选题、脚本或文章结构。
- 开发者：希望让工程师、架构师、测试、反方角色一起审查方案。
- 研究者：希望并行获得多个视角，再逐步互相交叉验证。

### 3.2 典型场景

#### 场景 A：独立专家评审

用户创建一个群聊，选择三个角色：

- 产品经理
- 工程师
- 反方

用户输入：

```text
分析一下这个产品想法：让多个 AI 像团队一样讨论一个问题。
```

系统分别把这条任务发给三个角色。三个角色互相不知道对方回答，只从自己的角色视角独立输出。

用户看到三份独立意见后，引用工程师的回复并 `@产品经理`：

```text
引用「工程师」的回复：……
@产品经理 你怎么看这个技术风险对产品路径的影响？
```

只有产品经理收到工程师观点和用户追问。

#### 场景 B：协作群聊讨论

用户创建一个“产品方案讨论”群聊，选择协作群聊模式。系统初始化时告诉每个角色群聊成员列表和各自职责。

第一轮用户发任务后，所有角色回答。第二轮用户继续问：

```text
如果两周内必须上线，怎么取舍？
```

系统发给每个角色时，会附带最近群聊上下文，包括其他角色上一轮发言。角色可以参考、补充或反驳其他成员观点。

## 4. 核心概念

### 4.1 群聊

群聊是一次讨论的容器。

字段建议：

```ts
interface GroupChat {
  id: string
  name: string
  mode: 'independent' | 'collaborative'
  roles: GroupRole[]
  messages: GroupMessage[]
  status: 'draft' | 'initializing' | 'ready' | 'running' | 'error'
  createdAt: number
  updatedAt: number
}

// 以上 roles/messages 是产品概念模型。
// 实际实现采用 normalized store：roleIds/messageIds + rolesById/messagesById。
```

产品上需要支持多个群聊同时存在。用户进入 OpenTeam 后看到的是一个类似微信的三栏结构：

```text
左侧：群聊列表
中间：当前群聊消息流
右侧：当前群聊角色列表和角色配置
```

左侧群聊列表展示所有已创建群聊，点击后切换当前群聊。

### 4.2 角色

角色代表群聊中的一个 AI 成员。这里需要区分“角色模板”和“群聊中的角色实例”。

- 角色模板：例如“产品经理”“工程师”“反方”，可以被多个群聊复用。模板包含 `name`、`description`、`systemPrompt`。
- 角色实例：某个群聊里的一个具体成员。即使两个群聊都用了“工程师”，它们也应该是两个不同的角色实例，拥有不同的 Gemini 会话、消息历史和上下文同步状态。

第一版必须支持角色模板库。用户可以创建角色模板；创建群聊或给群聊添加角色时，可以从模板库选择。模板加入群聊后会生成独立 `GroupRole` 实例，后续模板修改不自动影响已加入群聊的角色实例，除非用户手动同步/重新创建；第一版可以先不做模板同步能力。

因此，群聊内路由和持久化都不能只依赖通用角色模板 ID，而应该依赖当前群聊里的角色实例 ID。

字段建议：

```ts
interface GroupRole {
  id: string
  chatId: string
  templateId?: string
  name: string
  description?: string
  systemPrompt?: string
  status: 'pending' | 'loading' | 'ready' | 'thinking' | 'error'
  contextCursor?: number
  geminiConversationId?: string
  geminiConversationUrl?: string
  lastPromptMessageId?: string
  lastReplyAt?: number
  iframe?: {
    tabId: number
    frameId: number
  }
  createdAt: number
}
```

第一版角色字段至少需要：

- `id`
- `chatId`
- `name`
- `systemPrompt`
- `status`
- `contextCursor`
- `geminiConversationId`
- `geminiConversationUrl`
- `lastReplyAt`

`tabId` 和 `frameId` 是运行态，只能用于当前页面生命周期内的消息投递，不应该作为持久化恢复依据。

### 4.3 消息

消息包括用户消息、角色回复和系统消息。

字段建议：

```ts
interface GroupMessage {
  id: string
  chatId: string
  seq: number
  type: 'user' | 'assistant' | 'system'
  content: string
  roleId?: string
  roleName?: string
  targetRoleIds?: string[]
  references?: MessageReference[]
  createdAt: number
  status?: 'pending' | 'sent' | 'received' | 'error'
  deliveryStatus?: Record<string, 'pending' | 'sent' | 'received' | 'error'>
}

interface MessageReference {
  messageId: string
  roleId?: string
  roleName?: string
  contentSnapshot: string
}
```

`type` 统一使用 `'user' | 'assistant' | 'system'`，其中 `assistant` 表示群聊角色的 AI 回复。引用使用 `contentSnapshot`，避免原消息后续变化影响已经发送的上下文。

### 4.4 角色会话绑定

每个群聊角色实例需要绑定一个 Gemini 网页会话。这个绑定由两部分组成：

```ts
roleSessionKey = `${chatId}:${roleId}`
```

持久化绑定：

```ts
interface RoleConversationBinding {
  chatId: string
  roleId: string
  geminiConversationId?: string
  geminiConversationUrl?: string
  updatedAt: number
}
```

运行态绑定：

```ts
interface RuntimeFrameBinding {
  chatId: string
  roleId: string
  tabId: number
  frameId: number
  ready: boolean
}
```

恢复群聊时，系统用 `chatId + roleId` 找到该角色实例，再读取持久化的 `geminiConversationUrl`。如果存在历史 URL，就用该 URL 恢复 iframe；如果不存在，才打开新的 Gemini 首页。

不能只用 `roleId -> iframe` 做全局绑定，因为不同群聊可能复用同一个角色模板。正确的绑定粒度是：

```text
群聊 ID + 群聊内角色实例 ID + Gemini 会话 ID/URL + 当前运行态 frameId
```

## 5. 功能需求

## 5.1 创建群聊

### 描述

用户可以创建多个群聊，并设置群聊名称、群聊模式和角色成员。

### 入口

OpenTeam 团队页左侧群聊列表顶部提供“新建群聊”按钮。

### 创建流程

1. 用户点击“新建群聊”。
2. 输入群聊名称。
3. 选择群聊模式：
   - 独立专家模式
   - 协作群聊模式
4. 添加角色：
   - 从已有角色模板中选择。
   - 或直接创建新角色并保存为角色模板。
5. 点击“创建并初始化”。
6. 系统创建群聊记录，并为每个角色创建 Gemini iframe。
7. 所有角色准备完成后，群聊进入 `ready` 状态。

### 群聊列表

左侧群聊列表需要展示：

- 群聊名称
- 最近一条消息摘要
- 更新时间
- 当前群聊高亮状态
- 群聊状态，例如初始化中、运行中、异常

第一版可以先不做复杂搜索和分组，但必须支持多个群聊的创建、展示、切换和本地持久化。

### 验收标准

- 用户可以成功创建多个群聊。
- 用户可以在左侧列表中切换不同群聊。
- 用户刷新页面或重新打开插件后，群聊列表仍然保留。
- 用户可以在创建时添加至少一个角色。
- 用户可以选择独立专家模式或协作群聊模式。
- 群聊创建后，页面展示群聊名称、模式、角色列表和消息区。

## 5.2 创建角色

### 描述

用户可以在创建群聊时直接创建角色，也可以在群聊内继续添加角色。用户进入某个群聊后，可以在右侧角色区查看和编辑当前群聊中的每个角色配置。

### 角色模板与角色字段

第一版必须支持角色模板库，并支持从模板库创建群聊角色实例。

角色模板字段：

- `name`：模板名称，必填。
- `description`：模板描述，可选。
- `systemPrompt`：角色指令，可选。

群聊角色实例字段第一版建议支持：

- 角色名称，必填，同一群聊内必须唯一。
- 角色描述，可选。
- 角色指令，可选。

角色名称规则：不能包含空白字符，不能包含 `@`，不能是 `all`（大小写不敏感），避免与 `@all` 冲突。

示例：

```text
名称：工程师
描述：关注技术实现、复杂度、性能和稳定性。
角色指令：你是一名资深工程师，请从实现路径、技术风险和上线成本角度分析问题。
```

### 验收标准

- 用户可以创建角色模板，也可以从模板库选择角色加入当前群聊。
- 角色模板包含名称、描述和角色指令。
- 角色名称不能为空，并满足同群聊唯一、无空白字符、无 `@`、不能是 `all` 的规则。
- 新角色会显示在群聊角色列表中。
- 新角色加入后会自动初始化 Gemini iframe。
- 用户点击右侧角色后，可以编辑该角色的名称、描述和初始化指令。
- 修改角色配置后，仅影响当前群聊中的该角色，不自动重置会话。
- 用户可以点击“初始化/重新初始化”按钮，让当前 iframe 再发送一次初始化 prompt。
- 用户刷新页面或重新打开插件后，角色配置和角色模板库仍然保留。

## 5.3 初始化角色

### 描述

初始化角色时，系统为每个角色打开一个 Gemini iframe，并发送角色初始化 prompt。新角色第一次创建 Gemini 会话时，必须发送一次初始化 prompt；初始化回复需要写入群聊消息流，让用户能看到角色已完成初始化。初始化完成后，角色进入 `ready`。

如果角色已经有 `geminiConversationUrl`，恢复时不自动重新初始化。用户修改角色提示词后，右侧角色卡旁提供“初始化/重新初始化”按钮；点击后向该角色当前 iframe 再发送一条初始化 prompt，回复同样进入群聊消息流。编辑提示词本身不自动重置会话。

### 独立专家模式初始化

角色只知道自己的身份，不知道其他角色。

初始化 prompt 示例：

```text
你是「工程师」。

你的职责：
从技术实现、复杂度、性能、稳定性和上线风险角度分析问题。

用户会给你任务。请始终保持你的角色视角，独立回答，不需要假设还有其他 AI 成员。
```

### 协作群聊模式初始化

角色知道自己在群聊中，也知道群聊成员。

初始化 prompt 示例：

```text
你正在一个 AI 群聊中。

群聊成员：
- 产品经理：关注用户价值、市场需求和商业闭环。
- 工程师：关注技术实现、复杂度、性能和稳定性。
- 反方：负责指出漏洞、反例和失败风险。

你的身份是「工程师」。

请保持你的角色视角。你可以回应、补充或反驳其他成员的观点。当用户引用某位成员的发言时，请明确回应那条观点。
```

### 验收标准

- 点击初始化后，每个角色都创建一个 iframe，并发送一次初始化 prompt。
- 初始化回复写入群聊消息流。
- 每个 iframe 完成 content script 注册且初始化完成后，角色状态变成 `ready`。
- 有 `geminiConversationUrl` 的角色恢复时不自动重新初始化。
- 用户点击“初始化/重新初始化”后，会向当前 iframe 重新发送初始化 prompt，并把回复写入消息流。
- 初始化失败时，角色显示错误状态。
- 协作模式下，初始化 prompt 包含群聊成员列表。
- 独立模式下，初始化 prompt 不包含其他角色信息。

## 5.4 群聊发言与消息路由

### 默认路由规则

群聊中的发言默认发给全体角色。

规则：

```text
无 @：发给全部角色
@A：只发给 A
@A @B：只发给 A 和 B
```

`@all` 第一版不作为重点功能。可以兼容用户手写并路由给全体角色，但不在 UI 中主动展示。

手写 `@` 解析采用最长匹配，解决“产品”和“产品经理”同时存在时 `@产品经理` 应匹配“产品经理”的问题。角色名称规则见 5.2：同一群聊内唯一，不能包含空白字符，不能包含 `@`，不能是 `all`（大小写不敏感）。

### 独立专家模式路由

无 `@` 时：

- 同一条用户消息分别发送给所有角色。
- 每个角色只收到自己的角色指令和用户当前消息。
- 不自动携带其他角色的回复。
- 不额外传递该角色自己的历史，因为 Gemini 网页会在自己的会话里保留历史上下文。

有 `@` 时：

- 只发送给被点名角色。
- 如果用户附带引用，则把引用内容一并发送。

### 协作群聊模式路由

无 `@` 时：

- 同一条用户消息发送给所有角色。
- 每个角色收到自己尚未同步过的群聊上下文。
- 上下文包包含该角色上次收到消息之后，群聊里新增的其他成员发言和用户消息。

有 `@` 时：

- 只发送给被点名角色。
- 被点名角色仍会收到自己尚未同步过的群聊上下文。
- 如果用户附带引用，引用内容在 prompt 中重点展示。

### 验收标准

- 无 `@` 消息会发给所有角色。
- `@A` 消息只发给 A。
- `@A @B` 消息只发给 A 和 B。
- 独立模式不会自动携带其他角色发言。
- 协作模式会携带最近群聊上下文。

## 5.5 @ 选择器

### 描述

用户在输入框中输入 `@` 时，弹出角色选择列表。

### 交互

1. 用户输入 `@`。
2. 输入框上方或光标附近展示角色列表。
3. 用户可以用鼠标点击或键盘选择角色。
4. 选择后插入 `@角色名`。
5. 用户可以继续输入内容。

### 列表内容

第一版只展示具体角色：

```text
产品经理
工程师
反方
```

不默认展示 `@all`。

### 验收标准

- 输入 `@` 后能看到角色列表。
- 选择角色后能插入 `@角色名`。
- 支持一个消息中选择多个角色。
- 删除 `@角色名` 后，路由随之变化。

## 5.6 引用消息

### 描述

用户可以引用群聊中的某一条消息，并把引用内容发给指定角色。第一版只支持单条引用，不支持一次引用多条。

### 交互

每条角色消息旁提供“引用”按钮。点击后：

- 输入框进入引用状态。
- 显示被引用角色和引用摘要。
- 用户可以继续输入问题。
- 用户可以使用 `@角色` 指定回应者。

示例输入：

```text
引用「工程师」：
这个方案两周上线风险很高，因为认证和 iframe 稳定性都需要验证。

@产品经理 你怎么看这个风险对产品节奏的影响？
```

### Prompt 包装

发送给产品经理时，系统构造：

```text
你是「产品经理」。

用户引用了「工程师」的观点：
「这个方案两周上线风险很高，因为认证和 iframe 稳定性都需要验证。」

用户希望你回应：
你怎么看这个风险对产品节奏的影响？

请结合你的角色视角回复。请明确说明你同意或不同意哪里，以及下一步建议。
```

### 路由规则

- 如果引用后有 `@A`，只发给 A。
- 如果引用后没有 `@`，默认发给全体角色。
- 独立专家模式下，引用是跨角色信息传递的主要方式。
- 协作群聊模式下，引用内容会作为高优先级上下文。

### 验收标准

- 用户可以引用任意一条角色消息。
- 同一条待发送消息最多只能引用一条群聊消息。
- 引用状态能展示被引用角色和摘要。
- 用户可以取消引用。
- 发送后，目标角色收到引用内容。
- 回复消息能展示“回复/引用了谁”。

## 5.7 群聊上下文构造

### 独立专家模式

默认上下文：

```text
角色身份
用户最新消息
显式引用内容，如果有
```

不会自动包含其他角色回答。

独立专家模式不需要额外传递任何历史消息。每个角色对应的 Gemini iframe 本身已经拥有自己的会话历史，系统只需要把用户最新输入发过去。如果用户显式引用了某条消息，则额外带上这条引用内容。

### 协作群聊模式

默认上下文：

```text
角色身份
群聊成员列表
最近群聊上下文
显式引用内容，如果有
用户最新消息
回复要求
```

协作模式的核心不是每次重复携带最近固定几条消息，而是给每个角色维护一个“上下文同步游标”。

每条群聊消息都有递增序号：

```ts
message.seq = 1, 2, 3, 4...
```

每个角色记录自己已经同步到哪条群聊消息：

```ts
role.contextCursor = 12
```

含义是：这个角色已经收到过 `seq <= 12` 的群聊上下文。

当系统准备给某个角色发送新消息时，先找出这个角色还没收到过的群聊消息：

```ts
unsyncedMessages = messages.filter(message =>
  message.seq > role.contextCursor &&
  message.id !== userMessage.id &&
  message.roleId !== role.id
)
```

然后把这些消息作为“你上次之后，群聊里有这些新内容”附加到 prompt 中。这里必须排除当前正在发送的 `userMessage`，避免当前用户消息同时出现在“新增上下文”和“用户最新消息”里。content script 完成输入并触发 Gemini 发送后返回 ACK；background 收到 ACK 后，才把该角色的 `contextCursor` 更新到当前最新消息序号。如果发送失败或未收到 ACK，不更新 cursor。

示例：

```text
1 用户：分析这个方案
2 A：产品角度……
3 B：技术角度……
4 C：风险角度……
```

如果 A 的 `contextCursor = 2`，下一次发消息给 A 时，系统会带上：

```text
你上次之后，群聊里有这些新内容：

B：技术角度……
C：风险角度……

用户最新问题：
如果两周内上线，怎么取舍？
```

发送成功后，A 的 `contextCursor` 更新到最新消息序号。

这里需要注意：给 A 同步上下文时，不需要把 A 自己的发言作为新增上下文发给 A，因为 A 自己的 Gemini iframe 会话里已经有自己的历史。

### 上下文长度兜底

上下文同步游标解决“不要重复发送已经同步过的内容”的问题。但如果某个角色很久没有被点名，未同步消息可能很多，仍然需要长度兜底。

第一版采用如下兜底策略：

- 优先按 `contextCursor` 找未同步消息。
- 如果未同步消息过长，只发送最新 `maxContextChars` 字符，默认 6000 字。
- 超出时从较早的未同步消息开始截断。
- prompt 中必须明确说明“部分早期上下文已省略”。
- content script 返回 ACK 后，`contextCursor` 推进到当前最新消息序号。
- 第一版接受被截断的旧上下文不再逐条补发，后续版本再考虑摘要。

这不是用户可见的聊天记录限制，只是“每次发给角色时携带多少群聊上下文”的限制。用户在界面上仍然可以看到完整群聊记录。

### 验收标准

- 系统能根据群聊模式生成不同 prompt。
- 协作模式按角色增量同步未发送过的群聊上下文。
- 协作模式构造未同步上下文时，排除当前 `userMessage`。
- 已经同步给某个角色的上下文，不会在下一次重复发送给该角色。
- 给某个角色同步上下文时，默认过滤掉该角色自己的发言。
- 独立模式不包含其他角色最近发言，除非用户显式引用。
- 超长上下文会被截断。

## 5.8 角色会话恢复与 iframe 绑定

### 描述

每个群聊里的每个角色，都应该对应一个可恢复的 Gemini 会话。用户刷新页面、关闭再打开插件、或重启浏览器后，再回到该群聊时，系统应尽量通过已保存的 Gemini 会话 URL 恢复该角色，而不是无条件创建新会话。

需要明确 iframe 生命周期：关闭 Team 页面后，页面内所有 Gemini iframe 直接关闭，background 不再期望收到这些 iframe 的回复。只保证 Team 页面打开期间，已激活群聊的隐藏 iframe 可以继续生成并上报回复。

### Gemini 会话 ID

Gemini 网页 URL 中通常包含业务会话 ID。例如：

```text
https://gemini.google.com/app/<conversation-id>
```

这个 `<conversation-id>` 需要被提取并保存到角色实例上：

```ts
role.geminiConversationId = '<conversation-id>'
role.geminiConversationUrl = 'https://gemini.google.com/app/<conversation-id>'
```

content script 需要在以下时机上报当前 URL：

- iframe 初次 ready 时。
- Gemini URL 从首页跳转到具体会话页时。
- 发送 prompt 后 Gemini 创建新会话 ID 时。
- 页面发生 history pushState / replaceState / popstate 时。

background 收到后，按 `chatId + roleId` 更新本地存储中的 `geminiConversationId` 和 `geminiConversationUrl`。

### 初始化 iframe 时的 URL 选择

创建或恢复角色 iframe 时：

```text
如果 role.geminiConversationUrl 存在，且以 https://gemini.google.com/ 开头：
  iframe.src = role.geminiConversationUrl
否则：
  丢弃非法 URL，iframe.src = https://gemini.google.com/
```

这样用户恢复群聊时，角色能回到之前的 Gemini 会话页面。

### 绑定关系

持久化身份：

```text
chatId + roleId + geminiConversationId/geminiConversationUrl
```

运行态身份：

```text
chatId + roleId + tabId + frameId
```

`tabId` 和 `frameId` 只用于当前打开的 `team.html` 生命周期内投递消息。刷新后它们会变化，不能用于恢复。

### 多群聊切换与活跃 iframe

用户切换群聊时，可能出现以下情况：

- 当前群聊 iframe 已经渲染。
- 当前群聊 iframe 暂时没有渲染。
- 其他已激活群聊的角色还在隐藏宿主区生成回复。
- 页面刷新或关闭导致所有 iframe 生命周期结束。

要求：

- 在当前 Team 页面生命周期内，用户切换过/恢复过的活跃群聊，其角色 iframe 都保留在当前 Team 页面隐藏宿主区。
- 用户可以在这些活跃群聊之间切换，iframe 不销毁。
- Team 页面打开且某个群聊 iframe 已激活/隐藏保留时，background 必须能接收该群聊角色的回复。
- background 收到回复后，必须先写入 `chrome.storage.local`。
- 如果 `team.html` 当前打开且正在看对应群聊，再推送 UI 更新；如果当前正在看其他群聊，只保存数据，不丢消息。
- 关闭 Team 页面后，页面内所有 Gemini iframe 直接关闭，background 不再期望收到这些 iframe 的回复。
- 页面刷新/关闭视为 iframe 生命周期结束，后续需要按恢复流程重新创建。
- 用户后续打开对应群聊时，从本地记录渲染完整消息，并按需恢复 iframe。

### 恢复按钮

如果某个角色 iframe 关闭、加载失败或没有渲染，角色卡片提供“恢复角色”按钮。如果某个群聊没有活跃 iframe，群聊旁边提供“恢复聊天”按钮；点击后按该群聊角色的 `geminiConversationUrl` 重新创建 iframe。

点击恢复时：

1. 用 `chatId + roleId` 找到角色实例。
2. 读取 `geminiConversationUrl`。
3. 创建 iframe。
4. 如果有合法的 `geminiConversationUrl`，直接打开历史会话 URL；非法 URL 丢弃，不作为 iframe src。
5. iframe content script 注册新的 `frameId`。
6. background 更新运行态绑定。
7. 角色恢复为可发送状态。

### 验收标准

- 角色发送第一条消息后，系统能记录 Gemini 会话 URL。
- 刷新 OpenTeam 后，角色 iframe 使用历史 URL 恢复。
- 切换到其他群聊时，当前群聊消息不会丢失，已激活 iframe 保留在隐藏宿主区。
- Team 页面打开且某个群聊 iframe 已激活/隐藏保留时，background 收到的角色回复仍会先写入本地。
- 关闭 Team 页面后，background 不再期望收到页面内 iframe 的回复。
- 用户重新打开群聊后，可以看到之前收到的回复，并可通过恢复流程重新创建 iframe。
- `tabId/frameId` 变化不会影响持久化恢复。

## 5.9 角色忙碌状态与发送限制

### 描述

同一个角色在上一条消息还没有回复完成前，不能继续接收新的用户消息。否则 Gemini 同一个会话内会出现多个 prompt 交错，容易导致回复错乱或丢失。

### 规则

- 发送前检查所有目标角色：必须 `status === ready`，必须有 ready frame binding。
- 不允许向 `thinking`、`loading`、`error` 角色发送。
- 无 `@` 发给全体时，只要有一个目标角色不可用，整次发送阻止，不创建 user message。
- `@A` 时，如果 A 不可用，发送按钮需要阻止发送，并提示具体原因。
- content script 完成输入并触发 Gemini 发送后返回 ACK；background 收到 ACK 后，才把对应 `deliveryStatus` 标为 `sent` 并推进该角色 `contextCursor`。
- `tabs.sendMessage` 成功不等于投递成功。
- 角色回复完成后，状态回到 `ready`，并更新 `lastReplyAt`。
- 第一版不做单条失败消息重试。角色进入 `error` 时只提供“恢复角色/恢复聊天”，用户恢复后需要重新发送消息。

### 群发处理建议

无 `@` 群发时，如果任一目标角色不可用：

```text
只要有目标角色不是 ready，或缺少 ready frame binding，就阻止本次发送，并提示哪些角色需要等待或恢复。
```

这样能避免用户误以为所有人都收到了消息，也避免创建无法投递的 user message。

### 验收标准

- 只有 `ready` 且存在 ready frame binding 的目标角色可以接收消息。
- 角色生成中、加载中或错误时，用户不能继续向该角色发送消息。
- 群发时如果有目标角色不可用，系统阻止整次发送且不创建 user message。
- content script ACK 后才标记投递成功。
- 角色回复完成后，可以继续向该角色发送下一条消息。

## 8. 状态与异常

### 8.1 角色状态

```text
pending：已创建，未初始化
loading：iframe 加载中或初始化中
ready：可以接收消息
thinking：已收到 prompt，正在生成
error：出错
```

角色字段增加 `lastReplyAt?: number`，用于记录最近一次回复时间。业务上只有 `ready` 才表示可发送。

### 8.2 异常处理

- iframe 加载失败：显示“恢复角色/恢复聊天”按钮。
- Gemini 未登录或输入框不可用：角色进入 `error`，右侧角色卡显示“需要登录 Gemini”，提供“打开 Gemini 登录页”按钮；用户登录后点击“恢复角色/恢复聊天”。
- 发送失败：角色状态标为 `error`，并展示失败原因。
- 回复超时：角色状态标为 `error`，用户恢复后需要重新发送消息。
- 第一版不做单条失败消息重试。
- DNR 未生效：提示用户重新加载扩展，并提供诊断说明。

## 9. 数据与存储

第一版必须支持本地持久化。用户刷新 `team.html`、关闭再打开插件、或重启浏览器后，群聊、角色、消息、引用关系和上下文同步状态都必须保留。

推荐使用：

- `chrome.storage.local`：保存全部群聊数据、角色模板库、角色配置、消息记录、引用关系、上下文 cursor 和用户设置。
- `chrome.storage.session`：只用于临时运行态，例如当前 iframe 加载状态、临时错误提示等。

实际实现采用 normalized store：`roleIds/messageIds + rolesById/messagesById`。PRD 中 `roles/messages` 仅表达产品概念模型。

必须持久化字段：

- 群聊 ID
- 群聊名称
- 群聊模式
- 群聊列表排序
- 当前选中的群聊 ID
- 角色模板库
- 角色配置
- 角色初始化 prompt
- 角色所属群聊 ID
- 角色模板 ID，如果来自模板
- 角色 Gemini 会话 ID
- 角色 Gemini 会话 URL
- 消息记录
- 消息递增序号 `seq`
- 消息引用关系 `references`
- 消息投递状态 `deliveryStatus`
- 每个角色的 `contextCursor`
- 每个角色最近一次状态，至少包括 ready/error 等可恢复状态
- 用户偏好

不建议持久化字段：

- iframe 的 `tabId`
- iframe 的 `frameId`
- 临时 loading 状态
- 当前正在执行中的一次发送 Promise

这些运行态在页面重新打开后需要重新初始化。

### 9.1 恢复流程

用户重新打开插件时：

1. 从 `chrome.storage.local` 读取群聊列表、当前群聊 ID、角色、消息和 cursor。
2. 渲染左侧群聊列表。
3. 打开上次选中的群聊；如果没有选中记录，则打开最近更新的群聊。
4. 渲染该群聊的消息和角色配置。
5. 对该群聊中的角色按需重新创建 Gemini iframe；页面刷新/关闭后的 iframe 生命周期已经结束。
6. 如果角色有合法 `geminiConversationUrl`，iframe 直接打开该 URL；否则打开 Gemini 首页。
7. iframe 注册完成并上报 ready 后，将角色状态恢复为可发送状态；如果已有 `geminiConversationUrl`，恢复时不自动重新初始化。

### 9.2 写入时机

以下操作必须立即写入本地：

- 创建群聊。
- 修改群聊名称或模式。
- 切换当前群聊。
- 创建、编辑或删除角色。
- 用户发送消息。
- 角色回复消息。
- 创建或取消引用。
- 更新角色 `contextCursor`。
- 更新角色 `geminiConversationId`。
- 更新角色 `geminiConversationUrl`。
- 更新消息投递状态。
- 更新群聊消息序号。

### 9.3 存储失败处理

如果写入 `chrome.storage.local` 失败：

- UI 需要显示错误提示。
- 当前内存状态可以继续保留。
- 不应静默吞掉错误。
- 后续需要允许用户重试保存。

## 10. 成功指标

第一阶段验证指标：

- 用户可以成功创建群聊并初始化 2 到 4 个角色。
- 用户可以创建多个群聊，并在左侧列表中切换。
- 用户刷新页面、关闭再打开插件后，群聊、角色和消息记录仍然存在。
- 用户恢复群聊时，角色 iframe 能优先打开已记录的 Gemini 会话 URL。
- 用户无 `@` 发言后，所有角色都能收到并回复。
- 用户 `@A` 后，只有 A 回复。
- 用户引用 B 的消息并 `@A` 后，A 能围绕 B 的观点回复。
- 协作模式下，第二轮问题能看到角色参考尚未同步过的其他成员观点。
- 协作模式不会对同一个角色重复发送已经同步过的上下文。
- 某个角色正在回复时，用户不能继续向该角色发送下一条消息。
- Team 页面打开且该群聊 iframe 已激活/隐藏保留时，background 收到的回复仍会保存到本地。
- iframe 不出现持续刷新。
