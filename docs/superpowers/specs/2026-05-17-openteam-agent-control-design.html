一、目标

为 OpenTeam 增加本地智能体控制入口

外部智能体可以通过稳定的本地命令行工具控制 OpenTeam 浏览器扩展，完成创建群聊、添加人员、初始化网页 AI iframe、发布任务、等待回复、读取回复和导出结果。

创建群聊

由智能体提供结构化需求，OpenTeam 负责创建对应的群聊和上下文。

初始化人员

添加临时人员或人员库模板，并让站点 iframe 进入可投递状态。

发布任务

支持发给所有人员或指定人员，并在后台等待回复完成。

核心目标不是让智能体操作 OpenTeam 界面，而是新增一层本地控制平面。智能体调用结构化命令，OpenTeam 把命令转成已有群聊、人员、消息和编排能力。
二、背景与边界

复用已有能力，不重造群聊系统

OpenTeam 当前已经有成熟的内部能力，主要由 team.html 通过 chrome.runtime.sendMessage 调用。外部智能体无法直接调用这些运行时消息，因为它们运行在浏览器扩展权限域内。

已有内部能力

  • GROUP_CHAT_CREATE 创建群聊。
  • GROUP_ROLES_CREATE_BATCH 批量添加人员。
  • GROUP_MESSAGE_SEND 发布普通群聊消息。
  • GROUP_ROLE_RECOVER 恢复人员 iframe。
  • GROUP_ORCHESTRATION_RUN 运行编排流程。
  • GROUP_STORE_GET 获取完整 store。

第一版不做

  • 不让智能体直接点击 OpenTeam 界面。
  • 不暴露公网接口。
  • 不支持远程未认证访问。
  • 不把 OpenTeam 改成云服务。
  • 不要求用户配置模型 API key。
  • 不重写现有群聊、人员、iframe、消息投递架构。
  • 不强制支持完整编排流程生成和编辑。
参考 OpenCLI 的实现后，推荐采用“本地守护进程 + 扩展主动 WebSocket 连接”的模式，而不是要求用户用 --remote-debugging-port 启动 Chrome。

四、通信模型

HTTP 进入本地守护进程，WebSocket 进入扩展

命令行工具到本地守护进程

命令行工具通过本机 HTTP 发送命令，请求必须带 X-OpenTeam 和本地 bearer token。

POST /command
X-OpenTeam: 1
Authorization: Bearer <local-token>
Content-Type: application/json

扩展到本地守护进程

扩展 background service worker 主动连接本机 WebSocket，并在连接后发送 hello。

ws://127.0.0.1:<port>/ext

五、命令行功能

openteamctl 是智能体的主要入口

命令行工具默认输出 JSON，方便智能体读取。人类友好的表格输出可以后续再加。

核心命令

openteamctl doctor
openteamctl daemon status
openteamctl daemon start
openteamctl daemon stop
openteamctl daemon restart
openteamctl daemon logs

doctor 检查项

  • 本地守护进程是否可达。
  • 扩展是否已连接。
  • 扩展版本是否兼容。
  • OpenTeam 是否启用本机控制。
  • 是否可以打开或激活至少一个 team 页面。
  • 当前连接的 profile id。

群聊命令

openteamctl chat list
openteamctl chat get <chatId>
openteamctl chat create --name <name> --mode independent|collaborative
openteamctl chat delete <chatId>
openteamctl chat activate <chatId>

chat activate 只负责切换当前群聊并让 team 页面挂载相关 iframe，不发布任务。

人员命令

openteamctl role list --chat <chatId>
openteamctl role add --chat <chatId> --name <name> --site claude --prompt <text>
openteamctl role batch-add --chat <chatId> --file roles.json
openteamctl role recover --chat <chatId> --role <roleId>
openteamctl role wait-ready --chat <chatId> --timeout 120000

任务命令

openteamctl task post --chat <chatId> --target all --content <text>
openteamctl task post --chat <chatId> --target role:<roleId> --content <text>

openteamctl task stop --chat <chatId> --role <roleId>
openteamctl task retry --chat <chatId> --role <roleId> --message <messageId>

六、智能体技能

openteam-control 只做操作指南和薄适配

技能职责

  • 告诉智能体什么时候应该使用 OpenTeam。
  • 提供 create-and-post 的 JSON schema。
  • 要求智能体先运行 openteamctl doctor。
  • 新任务优先使用一键命令。
  • 已有群聊追问时使用 task post。
  • 定义结构化错误码的恢复策略。

技能不做什么

技能不承担业务逻辑，不直接和浏览器扩展通信，也不重写 OpenTeam 内部流程。真正的可靠性应该放在命令行工具、本地守护进程和扩展控制处理器里。

新任务流程 运行 doctor，生成 task.json，执行 run create-and-post --wait，返回回复后做总结。
超时流程 运行 task read --chat <chatId>，读取部分回复并说明哪些人员仍未完成。
追问流程 复用已有 chatId，执行 task post，等待并读取后续回复。
恢复流程 遇到 role_not_ready 或 role_recovery_failed 时，按错误码执行恢复或向用户报告需要手动处理。

界面需要增加一个开关：允许本机智能体控制 OpenTeam。说明文案必须明确风险：启用后，本机工具可以创建群聊，并通过用户已登录的 AI 网页会话发送 prompt。

Team 页面激活

部分操作必须依赖 team.html，因为 iframeHost 在页面里，不在 service worker 里。

CONTROL_TEAM_ACTIVATE_CHAT
  1. control handler 收到 chat.activate 或 run.createAndPost。
  2. background 确认 team 页面存在。
  3. 如果没有 team 页面，则打开 team.html。
  4. 扩展后台发送运行时消息给 team 页面。
  5. team 页面调用 iframeHost.activateChat。
  6. team 页面返回 role frame 状态。

人员初始化

激活群聊 先把目标群聊切到当前 team 页面。
恢复 iframe 对缺失或过期人员执行 GROUP_ROLE_RECOVER。
等待 ready 等待目标站点人员进入 ready，直到超时。
返回明细 返回 ready、pending、failed 的 role ids，方便智能体决策。 
interface InitializeChatResult {
  chatId: string
  readyRoleIds: string[]
  pendingRoleIds: string[]
  failedRoleIds: string[]
  timedOut: boolean
}

发布任务

长期推荐新增一个显式目标命令，避免依赖中文提及语法和人员名称。

GROUP_MESSAGE_SEND_TARGETED
{
  chatId: string
  content: string
  targetRoleIds: string[]
  reference?: MessageReference
}
如果想降低第一版改动量，也可以先由 task.post 合成 @所有人 或指定人员提及文本，再调用现有 GROUP_MESSAGE_SEND。但显式 target role ids 是更稳的长期设计。

等待回复

task.wait 应轮询 store 状态，不监听 DOM。

  • 所有目标人员 delivery status 为 received 时完成。
  • 任一目标人员为 error 时返回部分结果和错误。
  • 超时时返回部分结果。
  • 用户停止生成时返回 stopped 状态。
interface GroupMessage {
  sourceMessageId?: string
}

建议给 assistant message 增加 sourceMessageId，让 task.wait 和 task.read 能可靠找到对应回复。

八、本地守护进程设计

本地守护进程只做桥接，不理解业务

本地守护进程负责

  • 监听 127.0.0.1。
  • 提供 /ping、/status、/command、/logs、/shutdown。
  • 接受扩展在 /ext 的 WebSocket 连接。
  • 维护已连接扩展 profile。
  • 路由命令行工具命令到指定 profile。
  • 追踪 pending command id 和超时。
  • 记录审计日志。

本地守护进程不负责

  • 不直接访问 Chrome。
  • 不理解 OpenTeam store 内部细节。
  • 不解析或改写 prompt。
  • 不调用模型 API。
  • 不接管 team 页面 DOM。
默认端口 19826 环境变量 OPENTEAM_DAEMON_PORT 仅绑定 127.0.0.1

九、幂等设计

防止智能体重试造成重复创建和重复发送

智能体常常会在超时后重试。如果没有幂等机制，可能重复创建群聊、重复添加人员或重复发布任务。

需要支持 idempotencyKey 的命令

  • chat.create
  • roles.batchAdd
  • task.post
  • run.createAndPost

冲突规则

同 key 且同请求 hash，返回上次结果。同 key 但请求 hash 不同，返回 idempotency_conflict。

interface AgentControlIdempotencyRecord {
  key: string
  action: string
  requestHash: string
  result: unknown
  createdAt: number
  expiresAt: number
}

建议 TTL 为 24 小时。

十、安全设计

这是本地高权限自动化，默认必须关闭

必须加的保护

  • 控制功能默认关闭。
  • 本地守护进程只绑定 127.0.0.1。
  • HTTP 命令必须带 X-OpenTeam: 1。
  • HTTP 命令必须带本地 bearer token。
  • token 存在 ~/.openteam/control-token。
  • WebSocket 只接受 Chrome extension origin。
  • 正式版可校验固定 extension id。
  • 命令接口不返回 CORS 头。
  • 请求体限制 1 MB。

审计日志

写操作必须记录审计，但默认不保存完整 prompt，只保存 preview 和长度。

interface AgentControlAuditEntry {
  id: string
  action: string
  chatId?: string
  roleIds?: string[]
  messageId?: string
  promptPreview?: string
  promptLength?: number
  createdAt: number
  ok: boolean
  errorCode?: string
}

十五、待决问题

实现前需要确认的几个选择

openteamctl 放在哪里？

需要决定命令行工具是放在 OpenTeam 仓库里，还是拆成单独 package。最小可用版本可以先放在当前仓库，降低联调成本。

第一版是否增加 GROUP_MESSAGE_SEND_TARGETED？

显式 target role ids 更稳，但改动略大。mention 合成更快，但长期会受名称和语言影响。

task.wait 是否立即增加 sourceMessageId？

建议增加。它能让任务等待和读取结果更可靠，避免通过消息顺序推断。

端口是固定还是界面可配置？

建议默认固定为 19826，后续可用环境变量或设置项覆盖。

命令行工具是否自动启动本地守护进程？

建议学习 OpenCLI，浏览器相关命令自动启动本地守护进程，减少用户心智负担。

技能放在仓库里还是用户本地技能目录？

最小可用版本可以先随仓库维护，稳定后再考虑安装到用户本地技能目录或插件包。

推荐用 OpenCLI 风格的本地守护进程桥接方案作为基础。最小可用版本只实现普通群聊自动化：创建群聊、添加人员、激活和初始化、发布给所有人、等待并读取回复。完整编排控制等这一条链路稳定后再做。

这条路线能最大限度复用 OpenTeam 现有能力，同时给智能体一个稳定、结构化、可诊断、可恢复的本地工具接口。
